# Scoring Pipeline

The match agent (`agents/match_agent.py`) turns a raw list of scraped jobs into a ranked shortlist. It runs in three stages: hard pre-filters, a two-stage semantic recall + rerank, and a final score assembly.

---

## Stage 0 — Hard Pre-filters

Applied before any scoring. A job that fails any of these gets `score: 0` and is dropped immediately.

| Filter | Config key | Description |
|---|---|---|
| Excluded company | `exclude_companies` | Company name is in the blocklist |
| Excluded location | `exclude_locations` | Location string contains a blocked term (word-boundary matched for short codes like "US") |
| Excluded keyword | `keywords_exclude` | Job title or description contains a disqualifying keyword (e.g. PHP, .NET) |
| Experience gate | `experience_years.max` | Description mentions a hard requirement above your max years (regex: `5+ years`, `minimum 6 years`, etc.) |
| Salary floor | `min_salary_lpa` | A parseable salary in the listing is below your minimum |
| Remote region | `remote_locations` | Job is remote but the listed region is not in your allowed list |

---

## Stage 1 — BM25 Recall

All jobs are ranked by BM25 (Okapi BM25 via `rank-bm25`). The query is built from your preferences with weighted repetition:

- Must-have keywords → **3×** weight
- Tier-1 nice-to-haves → **2×** weight
- Tier-2 nice-to-haves → **1×** weight
- Target role titles → **1×** weight

Only the **top 150** jobs by BM25 score advance to Stage 2. The rest are dropped as below-threshold. BM25 scores are min-max normalized to **0–30**.

---

## Stage 2 — JobBERT Bi-Encoder Rerank

The top-150 jobs are re-scored using [`jjzha/jobbert-base-cased`](https://huggingface.co/jjzha/jobbert-base-cased), a BERT model fine-tuned on job postings.

**Candidate query** (built from `data/parsed_resume.json`):
```
Skills: Python, FastAPI, PostgreSQL, ...
Experience: 3 years, mid-level
Industries: Cybersecurity, SaaS
Role targets: Backend Engineer, SDE II
```

Each job is encoded as:
```
Job Title: <title>
Job Description: <first 1500 chars>
```

Cosine similarity between the candidate embedding and each job embedding is min-max normalized to **0–40**.

`data/parsed_resume.json` is generated by `utils/resume_parser.py` — run `just parse-resume <path>` to create or update it.

---

## Stage 3 — Score Assembly

The final score (0–100) is the sum of five components:

| Component | Range | Source |
|---|---|---|
| BM25 keyword match | 0–30 | Stage 1 normalized BM25 score |
| JobBERT semantic similarity | 0–40 | Stage 2 cosine sim score |
| Title match | 0–10 | Full role match = 10; partial (≥2 meaningful words) = 4 |
| Industry match | 0–10 | Any `industry_keywords` term or `industries` name found in posting |
| Location match | 0–10 | Remote, hybrid in preferred city, or exact location match |

> The title, industry, and location subscores are computed by `_score_job()` and then rescaled: title is divided by 2.5 before being added (max 10 instead of 25).

A job qualifies if `final_score >= min_match_score` (default: `50` in `preferences.yaml`).

---

## Output

Qualifying jobs are sorted descending by score. Each job gets a `match_analysis` dict attached:

```json
{
  "score": 74,
  "bm25_score": 22,
  "jobbert_score": 31,
  "title_score": 8,
  "industry_score": 10,
  "location_score": 10,
  "reasons": ["Strong title match", "3/3 must-have keywords matched", "Industry match: Cybersecurity"],
  "missing_keywords": [],
  "strengths": ["Python", "FastAPI", "REST API"]
}
```

This is stored in the database and surfaced in the dashboard.
